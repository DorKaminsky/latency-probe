What was the most difficult part of this assignment?
=====================================================

The hardest design decision was around concurrency and state management.

The assignment asks for a REST service that accepts a URL and an interval,
then continuously polls that URL at the given frequency. The natural
implementation is a background loop — but how do you run background loops
inside a web server cleanly?

The options I considered:

1. Thread per job (threading.Thread):
   Simple to understand, but CPython's GIL limits true parallelism for
   CPU-bound work, and thread safety around the _jobs dict requires locks.
   For I/O-bound polling (which this is), threads mostly just sleep and wake
   — acceptable but noisier than necessary.

2. Celery + Redis:
   Proper distributed task queue, survives restarts, supports multiple
   workers. Overkill for this scale — adds two infrastructure components
   (broker + result backend) and significant operational overhead for what
   is essentially a demo service.

3. asyncio.create_task (chosen):
   Each probe job is a coroutine scheduled on the same event loop that
   FastAPI/uvicorn uses. No threads, no extra processes, no external
   dependencies. The event loop multiplexes between serving HTTP requests
   and running probe tasks through cooperative multitasking — when a probe
   task calls `await asyncio.sleep(interval)` or `await client.get(url)`,
   the event loop is free to handle other work.

   The trade-off: state lives in the process's memory (_jobs dict). If the
   pod restarts, all running jobs are lost. For production, this would be
   addressed by persisting jobs to Redis or a database and rehydrating on
   startup — but that scope is beyond the assignment.

The second challenge was measuring latency accurately. Using
`time.perf_counter()` around a requests.get() call includes Python
interpreter overhead, connection pool acquisition, etc. Using
`httpx.Response.elapsed` (a timedelta set by the httpx library after the
full response body is received) gives a cleaner application-level view of
the server's response time, which is what the assignment actually asks for.

The third challenge was a security one I only noticed after building the
happy path: this service is a textbook SSRF (Server-Side Request Forgery)
vector. A caller can POST any URL — including http://169.254.169.254
(the AWS instance metadata endpoint), internal k8s service IPs, or
loopback addresses. An attacker who can reach the probe API can use it to
pivot inside the cluster network or extract cloud credentials.

The mitigation happens in two places, sharing one blocklist (RFC-1918,
loopback, link-local including 169.254.0.0/16, and IPv6 equivalents):

1. Pre-flight: the route handler calls `resolve_and_check(url)` before
   creating the job. This resolves the hostname via async
   `loop.getaddrinfo` (with a 3-second timeout so a hostile DNS server
   can't stall the API) and rejects the request if any resolved address
   falls in a blocked range.

2. Per-request: the probe loop uses a custom httpx transport
   (`_SSRFGuardTransport`) that re-resolves the hostname and re-checks
   the IP on every single request. This defends against DNS rebinding —
   an attacker registering a domain that resolves to a public IP during
   validation and flips to `169.254.169.254` on subsequent lookups.

Additionally, `follow_redirects=False` prevents a 302 from escaping the
check: without it, a target could redirect to an internal URL and bypass
the transport-level guard.
